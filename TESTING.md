# Testing

jobSearch is a headless, scheduled script with no UI, no HTTP API of its own, and no database — so this suite is shaped around what the project actually is, not a template borrowed wholesale from a web app. See `CONTEXT.md` for the full reasoning behind each layer's shape; this document tracks what's built versus what's still ahead.

## Status: full coverage across all five layers, 99.07% statement / 100% function coverage on check-jobs.ts

Every layer listed below has real, executed, passing coverage for every source/function it's meant to cover, including a full `main()`-level integration test, an `AI_PIPELINE_ENABLED=true` integration test, and a genuine fault-injection test proving `safely()`/`safelyValue()`/`safelyRun()` contain a real rejected fetch rather than crashing the run. All numbers in this document were confirmed by an actual `npm test` (198 Vitest tests, 35 files) and `npm run test:email` (5 Playwright tests, 2 files) run, not inferred from what should be true. `npm run test:coverage` reports 99.07% statements / 89.03% branches / 100% functions on `check-jobs.ts` — up from 96.09%/88.54%/100% before this round, now that both previously-known coverage gaps are closed.

### AI_PIPELINE_ENABLED test isolation

`AI_PIPELINE_ENABLED` is read into a module-level constant at import time (`const AI_PIPELINE_ENABLED = process.env.AI_PIPELINE_ENABLED === "true"`), not re-read per call. Setting `process.env.AI_PIPELINE_ENABLED` inside a `beforeEach` after `check-jobs.ts` has already been statically imported has no effect, since the constant is already frozen. `tests/integration/main-ai-pipeline.test.ts` works around this by setting the env var first, then using `vi.resetModules()` plus a dynamic `import("../../check-jobs.ts")` inside each test to force a fresh module evaluation that picks up the new value. Keeping this in its own file (rather than a second `describe` block bolted onto `main.test.ts`) keeps that reset from affecting the other, unrelated `main()` tests.

## Deliberately not included in this phase

- **Cucumber / BDD.** Considered and deliberately skipped. shenny's Cucumber layer earns its keep by mirroring an E2E suite that already exists and reusing its browser automation — a relatively cheap English-language layer on top of infrastructure that's already built. jobSearch has no equivalent browser automation to piggyback on, so a Gherkin layer here would be a parallel structure built from scratch, largely restating what well-named `describe`/`it` blocks already say in plain English, for an audience of one person who already reads TypeScript comfortably. Revisit only if a specific scenario comes up where Gherkin would add something a unit test genuinely can't.

## Real quirks discovered while building out this suite

Writing tests against real captured API responses (rather than invented fixtures) surfaced several genuine field-mapping gaps in the fetcher code. All of them are now fixed, each behind its own test-first regression test:

- **`fetchAshbyJobs` now maps `workArrangement`** from the real API's `workplaceType`/`isRemote` fields, which were present in the real captured fixture and simply never read. Previously every Ashby posting had `workArrangement: undefined` regardless of what the source actually said.
- **`fetchRemoteOKJobs` no longer throws on a non-array response.** It called `data.filter(...)` directly with no guard; if RemoteOK ever returns an error object (e.g. rate limiting) instead of an array, this would throw inside the fetcher rather than being caught gracefully the way `fetchLeverJobs`'s existing `Array.isArray` guard already handles the equivalent case. Now guarded the same way.
- **`fetchTherapyNotesJobs` now filters by title** via `matchesAnyTitle`, matching every other fetcher's pattern (TherapyNotes being an allowlisted company controls which board gets queried, not whether title-matching still applies to what comes back). Its `location` mapping now builds from the real top-level `city`/`state`/`country` fields (`job.location` doesn't exist on the real API) instead of always returning `undefined`. As of this fix, none of the 3 postings currently on the real captured board actually match `SEARCH_TITLES` — a real, current fact about that board, not a test artifact.
- **`fetchTitleSearchJobs` (the Workable cross-search source)** now maps `company` from `job.company.title` (the real field; `.name` doesn't exist), builds `location` from the real `{city, subregion, countryName}` object into a joined string (skipping empty/null parts), and maps `postedAt` from `job.updated` (the real field; `job.updatedAt` doesn't exist, kept only as a harmless fallback).
- **`daysOld` no longer misinterprets a RemoteOK-style epoch-seconds `postedAt` as milliseconds.** `job.epoch` (used when `job.date` is absent) is Unix seconds; `new Date(n)` always treats a numeric argument as milliseconds. The fix converts at the `fetchRemoteOKJobs` call site (`job.epoch * 1000`) rather than changing `daysOld`/`daysAgoLabel` themselves, since every other source already passes a proper ISO date string through them.
- **USAJOBS salary ranges now use `remuneration.Description`** (e.g. `"Per Year"`, `"Per Hour"`) instead of the raw `RateIntervalCode` abbreviation (e.g. `"PA"`). The real API already returns the human-readable field right next to the code; it's shaped correctly for `formatSalaryRange`'s existing `"Per X"` → `"/X"` conversion, the same way Adzuna's `"Per Year"` string already works.
- **`extractUnconfirmedTerms` no longer leaks the literal word "unconfirmed"** into its returned term set. The `### Unconfirmed` heading line is now stripped before tokenizing, inside `extractUnconfirmedTerms` itself — `significantWords` (shared with the template-kit matching engine) was left untouched.
- **`fetchGreenhouseJobs` now reads `job.content` into `description`**, so `yearsRequired`/`workArrangement` extraction actually runs on Greenhouse postings. A real live capture against `https://api.greenhouse.io/v1/boards/impiricus/jobs?content=true` confirmed the real shape: `job.content` is HTML markup that is itself HTML-entity-encoded (a literal `<p>` tag arrives as the text `&lt;p&gt;`), so a small local `stripGreenhouseContent` helper decodes entities first, then strips tags, then collapses whitespace. `tests/fixtures/greenhouse-response.json` was replaced with a fresh real capture that includes `content` for both jobs.
- **`stripHtml` was removed.** It was defined and exported but had zero call sites in `check-jobs.ts`'s actual control flow — dead code. The Greenhouse `content` gap above got its own small local, non-exported helper instead of reviving it.

## A note on coverage as a number

"Complete coverage" here means every meaningful behavior has a test that would fail if the behavior broke — not a 100% line-coverage target for its own sake. `npm run test:coverage` is there to spot obviously untested code, not to be chased as a score.
