# jobSearch — Testing CONTEXT.md

Framing rules for building out the test suite. Read this before writing any test.

## What this project actually is

`check-jobs.ts` is a single-file, headless, scheduled Node script — not a web app. It has:

- **No UI.** Nothing to click through. "E2E" doesn't mean browser flows here.
- **No HTTP API of its own.** It only _calls_ other people's APIs (Adzuna, USAJOBS, SOLTECH's RSS, Quarterhill's internal JSON API, etc.) and scrapes one HTML page (Statheros). There are no routes to test.
- **No database.** State is three JSON files committed back to the repo by CI: `seen-jobs.json`, `application-tracker.json`, `digest-state.json`, plus a hand-edited `company-history.json`.
- **One real visual artifact**: the generated email HTML. That's the only place a browser engine adds genuine value over string assertions.

Don't port shenny-test-showcase's five layers literally — they assume a running web app with routes, screens, and a Postgres/Prisma database. None of that exists here. The remapping below is deliberate, not a shortcut.

## The five layers, and why each one is shaped the way it is

1. **`tests/unit/`** (Vitest) — pure functions: `scoreJob`, `isRemoteJob`, `isFreshJob`, `formatSalaryRange`, `extractYearsRequired`, `extractWorkArrangement`, `dedupeBySignature`, `resolveProvider`, `sourceLabel`, `historyStatus`, `isBlockedCompany`, `daysOld`/`daysAgoLabel`, `slugify`, `isAllowlistedCompany`, `draftHeader`, the template-kit's matching engine (`buildTemplateDraft`, `buildCoverLetter`, `significantWords`, `extractUnconfirmedTerms`). This is jobSearch's real primary logic layer — the equivalent of shenny's "API contract tests, the layer that owns the bulk of logic coverage." This layer also ended up owning the AI-drafting pipeline (`callAiModel`, `prefilterJob`, `draftResume`, `fetchResumeCorpus`, `commitDraft`, `githubApi`) — a deliberate classification call, not an obvious fit: these functions do make real outbound network calls like the fetchers layer does, but they don't map anything into `JobPosting` shape, which is `tests/fetchers/`'s actual defining trait per this document. They're closer in spirit to "given a mocked network layer, assert this utility's branching/output," which is what the rest of `tests/unit/` already does — so mocked-`fetch` tests for them live here instead of in `tests/fetchers/`. Revisit this call if the AI pipeline grows enough to justify its own layer. **Status: built** — 17 files, 122 tests, all passing (see `TESTING.md`).
2. **`tests/fetchers/`** (Vitest, mocked `fetch`) — one file per source, fed a realistic fixture response, asserting correct field mapping into `JobPosting`. This is the closest thing to shenny's "API contract" layer, just aimed at outbound calls instead of inbound routes. **Status: built** — all 11 sources covered, 54 tests passing. 10 of 11 fixtures are real captured API responses; Lever's is built from Lever's documented public API shape since `LEVER_COMPANIES` is currently empty (no live board to capture from) — see the note in `tests/fetchers/lever.test.ts` and revisit with a real capture if a company is ever added there. Writing these against real data surfaced several genuine field-mapping quirks in the fetchers themselves, all now fixed test-first (see `TESTING.md`'s "Real quirks discovered" section) rather than inventing clean fixtures that would have hidden them. Most sources also cover the "malformed/empty response doesn't throw" branch.
3. **`tests/state/`** (Vitest, real temp files) — `loadSeenJobs`/`saveSeenJobs`, `loadTracker`/`saveTracker`, `loadDigestState`/`saveDigestState` + `isDigestSource`, `loadCompanyHistory`. Test against real files on disk in a real temp directory, not mocked `fs` — this is exactly the class of bug that bit the project twice already (the `git-auto-commit-action` pathspec failure, caused by a file that silently never got created). Don't call this a "DB layer" — there's no database, and calling it one will mislead whoever reads it next. **Status: built** — 4 files, 15 tests passing.
4. **`tests/integration/`** — `pipeline.test.ts` composes the real filter → dedupe → score pipeline on a realistic mixed-source array (the two real bugs this project has actually shipped, `isRemoteJob`'s one-directional signal gap and the SQL false-positive gap-flag, both lived exactly at this composition boundary, not inside any single function). `main.test.ts` is the full `main()`-level integration test — all 11 network sources, Resend, and the GitHub API (for `resume-vault` reads/writes) mocked behind a single URL-routed `fetch` mock, covering a normal run (immediate alert + digest queued but not sent), a first run (baseline only, no side effects), a later run where a previously-queued digest actually sends, and a fault-injection run proving `safely()` contains a genuinely rejected fetch call from one source while the other 10 still come through. `main-ai-pipeline.test.ts` is a dedicated file covering the `AI_PIPELINE_ENABLED=true` path end-to-end (successful draft+commit+email-attachment case, and the prefilter-rejects case) — see the "Real gotchas" section below for why it needs its own file rather than a `describe` block inside `main.test.ts`. **Status: built** — 3 files, 7 tests passing.
5. **`tests/email/`** (Playwright) — the one legitimate use of a browser here. `buildAlertEmailHtml`/`buildDigestEmailHtml` were extracted specifically to make this possible without mocking the Resend SDK or sending a real email. Render the real HTML string via `page.setContent()`, assert on real DOM. Don't add more Playwright usage beyond this — there's no other visual surface in this project to justify it. **Status: built and actually executed** (not just typechecked) — `npx playwright install chromium && npm run test:email` runs clean, 2 files, 5 tests passing.

**Deliberately not built yet:**

- **Cucumber/BDD.** Deliberately skipped — see HANDOFF.md for the reasoning. Only reconsider if a specific case comes up where a Gherkin scenario would genuinely add something a well-named Vitest `describe`/`it` block doesn't already say in plain English.

## Real gotchas discovered while scaffolding this — don't rediscover these the hard way

- **`new Resend(process.env.RESEND_API_KEY)` runs at module load and throws on a missing key** — not just stores `undefined` silently. Any test importing from `check-jobs.ts` needs `RESEND_API_KEY` set, even if the test never touches email. Handled via `vitest.config.ts`'s `test.env`. If you add other module-level side effects to `check-jobs.ts` later, check whether they need the same treatment.
- **`main()` is guarded**: `if (process.argv[1] === fileURLToPath(import.meta.url)) { main(); }`. Note this uses `fileURLToPath` rather than a raw string comparison against `import.meta.url` — a raw `file://${process.argv[1]}` comparison silently never matches on Windows (slash direction, `file://` prefix), so don't simplify it back to that. This is why importing the module for tests doesn't trigger a live run. Don't remove this guard.
- **`loadCompanyHistory()` runs at module load too** (`const COMPANY_HISTORY = loadCompanyHistory();`), reading `company-history.json` relative to `process.cwd()`. It's safe — the function has a try/catch fallback to an empty Map — but it means tests that run from the repo root will pick up the _real_ `company-history.json`. `company-history.json` uses a richer per-company schema (`applications[]`, `aliases[]`, `status: "active" | "caution" | "blocked"`, etc. — see the type exports in `check-jobs.ts`) rather than a flat name→string map; `status` is the field that actually drives behavior (`"caution"` deprioritizes/tags like the old single-status scheme did, `"blocked"` — new — removes the job from the pipeline entirely via `isBlockedCompany`, before it's ever scored). Both real entries in the repo file (Golden Pet Brands, TherapyNotes) are currently `"active"` by design, since neither history warrants a flag yet — see `historyStatus`/`isAllowlistedCompany`/`isBlockedCompany`/`scoreJob`'s test files for how the `"active"` cases lean on the real file while `"caution"`/`"blocked"` are exercised against a synthetic file in an isolated temp cwd plus a fresh module import (`vi.resetModules()` + dynamic import, since `COMPANY_HISTORY` is frozen at first import same as `AI_PIPELINE_ENABLED` below). Keep leaning on the real file for the `"active"`/no-entry cases unless there's a specific reason to isolate a test from real repo state.
- **`TRACKER_PATH`/`SEEN_JOBS_PATH`/`DIGEST_STATE_PATH` are hardcoded relative paths, not injectable.** State-layer tests use `process.chdir()` into a fresh `mkdtempSync` temp directory for each test (see `tests/state/tracker.test.ts`) rather than mocking `fs`. This is deliberate — it tests the real read/write functions against a real filesystem, which is the whole point of this layer, while never touching the actual repo's state files.
- **Playwright's browser binary is now confirmed installable and runnable** (`npx playwright install chromium && npm run test:email` runs clean) — the earlier note about the scaffolding sandbox not reaching `cdn.playwright.dev` no longer applies in an environment with normal network access. All 5 email tests have actually been executed and pass, not just typechecked.
- **Test titles must genuinely match `SEARCH_TITLES`.** A test job titled something that sounds plausible but isn't a real substring match (e.g. "QA Support") will silently get filtered out by `matchesAnyTitle` and the test will fail in a confusing way. Check the actual `SEARCH_TITLES` array before picking a fixture title. As of the real-fixture fixes below, none of the 3 postings currently on TherapyNotes' real captured board are substring matches for `SEARCH_TITLES` either — a real, current fact about that board, not a bug.
- **`numeric postedAt` values are treated as milliseconds, not seconds, by `daysOld`/`daysAgoLabel`** — this is unchanged and intentional, since every real source passes a proper ISO date string through them. RemoteOK's `job.epoch` fallback (used when `job.date` is absent) is Unix seconds, so it would land near 1970 if passed straight through; fixed at the `fetchRemoteOKJobs` call site instead by converting to milliseconds (`job.epoch * 1000`) before it ever reaches `daysOld`/`daysAgoLabel`. See `tests/fetchers/remoteok.test.ts`.
- **All previously-documented fetcher field-mapping mismatches are now fixed** (TherapyNotes' `location`/title filter, Workable cross-search's `company`/`location`/`postedAt`, USAJOBS' salary interval suffix, Greenhouse's missing `content` mapping) — each behind its own test-first regression test grounded in a real captured API response. See `TESTING.md`'s "Real quirks discovered" section for the full list and what each fix actually does.
- **Greenhouse's real `job.content` field is HTML markup that is itself HTML-entity-encoded** — a literal `<p>` tag arrives in the JSON as the text `&lt;p&gt;`, confirmed by a real live capture against `https://api.greenhouse.io/v1/boards/impiricus/jobs?content=true`. Decode entities *before* stripping tags, not after, or the tag-strip regex finds nothing to strip. See the local `stripGreenhouseContent` helper in `check-jobs.ts`.
- **A field-mapping test asserting "the whole real fixture returns `[]`" goes stale the moment the underlying code (or `SEARCH_TITLES`) changes to correctly match one of that fixture's postings.** `tests/fetchers/therapynotes.test.ts` had exactly this: a test asserting the real captured board fully filters out, written before `SEARCH_TITLES` gained a `"Quality Assurance Engineer"` entry that now matches the fixture's first job. The test kept "passing" as a description in prose but was actually asserting the pre-fix, buggy behavior — nothing failed until the underlying code changed and the assertion caught up. When a "verify this is fixed" task claims a bug is unresolved, always run the suite for real before assuming the fixture/test pair still encodes that claim; a stale assertion masquerading as a regression test is the same failure shape as the bugs themselves (silently wrong, no crash).
- **SOLTECH's RSS feed carries real per-item location signal in namespaced XML tags outside `<description>`** (`job:locationCity`, `job:locationState`, `job:locationCountry`), confirmed by a live capture of `https://soltech.hire.trakstar.com/jobfeeds/soltech`. `extractXmlTag(item, "job:locationCity")` works fine on a namespaced tag name since the colon is just a literal character in the constructed regex. **`job:locationCountry` is not reliable** — the real capture had it holding a zip code (`"30326"`), not a country name, a data-quality quirk in SOLTECH's own feed. Only `city`/`state` are mapped into `location`; `country` is deliberately left out.
- **Quarterhill's `job.description` field exists and is real** (a previous audit wrongly assumed it didn't) — but scanning it with the shared `extractWorkArrangement()` is unsafe for this source specifically. A real live posting explicitly marked remote via `location_name: "Remote - US"` has description body copy about being "required to work in all weather conditions, travel to/from various **remote locations**" alongside an unrelated "acceptance **onsite**" phrase — `extractWorkArrangement` checks `onsite` before `remote`, so scanning the description alone would have flipped a job Quarterhill's own structured field says is remote into `"onsite"`, and `isRemoteJob` would have silently dropped it. Fixed by checking `location_name` for the literal word "remote" first, only falling back to `extractWorkArrangement(description)` when `location_name` doesn't decide it. Don't assume a source's free-text description is a safe signal just because other sources' descriptions are — check what the real text actually says before wiring a shared extractor in.
- **Workable's cross-search endpoint (`jobs.workable.com/api/v1/jobs`) has a `job.workplace` field** (`"remote"` / `"hybrid"` / `"on_site"`, confirmed via a live multi-query capture) that the account-widget endpoint (`apply.workable.com/.../widget/accounts/...`) doesn't have — that one only has the `telecommuting` boolean. Two different endpoints from the same vendor, two different remote signals; don't assume one implies the other exists.
- **Lever's `workplaceType` field (`"remote"`/`"hybrid"`/`"on-site"`) is documented-shape only, not live-verified** — `LEVER_COMPANIES` is still empty, so there's no board to capture from. Mirrors Ashby's `workplaceType` handling. Revisit with a real capture the day a company is added.
- **A systematic title-coverage sweep** (querying every `SEARCH_TITLES` entry against `jobs.workable.com`'s cross-search endpoint and diffing all ~240 unique titles returned against `matchesAnyTitle`) found real, current, QA/SDET-adjacent postings missing purely because of phrasing gaps, not typos: `"Software Engineer in Test"` (missing "Development"), `"Software Tester"` and `"Software Testing Engineer"` (gerund form, not a substring of "Test Engineer"), `"QA Architect"`, `"Test Automation Specialist"`, `"Automation Testing Engineer"`. All six added to `SEARCH_TITLES`. Some real titles are still deliberately unmatched — e.g. `"Software Quality Assurance - Engineer"` and `"Journeyman Software Quality Assurance Tester-Engineer"` — because the gap there is inserted punctuation/words breaking a substring match, not missing vocabulary; chasing every such variant with narrower and narrower phrase entries doesn't converge. See `matchesAnyTitle.test.ts` for both the fixed cases and the documented-not-fixed ones.
- **`isRemoteJob` never checked geography — only whether a posting was remote at all, not remote from where.** Workable's cross-search endpoint (`jobs.workable.com/api/v1/jobs`, `fetchTitleSearchJobs`) has no country scoping in the query — it searches every company on Workable worldwide — and `isRemoteJob` unconditionally trusts `workArrangement === "remote"` regardless of location. `tests/fixtures/workable-search-response.json` is a real capture containing exactly this: a job with `workplace: "remote"` and `location.countryName: "Peru"`, which the pre-fix suite asserted passed through cleanly. RemoteOK postings are worse — they bypass `isRemoteJob` entirely (`if (job.key.startsWith("rok:")) return true;`), so a RemoteOK "Remote (EU only)" posting had no geographic check anywhere in the pipeline. Fixed with a `country?: string` field on `JobPosting` (real structured value from TherapyNotes/Workable, hardcoded `"United States"` for Adzuna since its query is already `/us/`-scoped) and a new `isDomesticJob()` filter in `main()`'s chain: trusts `job.country` when present, otherwise scans `location` text against a `NON_US_LOCATION_KEYWORDS` denylist for sources with no structured country field. This denylist approach is a pragmatic middle ground, not exhaustive — a foreign posting whose location string is just `"Remote"` with no country name or region marker at all will still get through. See `tests/unit/isDomesticJob.test.ts`.
- **`AI_PIPELINE_ENABLED` is a module-level constant, not a value re-read per call.** `const AI_PIPELINE_ENABLED = process.env.AI_PIPELINE_ENABLED === "true"` is evaluated once, at `check-jobs.ts`'s first import. Setting `process.env.AI_PIPELINE_ENABLED` inside a test's `beforeEach` has no effect on a statically-imported copy of the module, since the constant is already frozen by then. `tests/integration/main-ai-pipeline.test.ts` works around this with `vi.resetModules()` plus a dynamic `import("../../check-jobs.ts")` inside each test, after setting the env var — and lives in its own file so that reset doesn't affect `main.test.ts`'s other tests.

## Workflow rule going forward

Every change to `check-jobs.ts` gets a test written before the change is considered done — not after, not "later." If a function doesn't have a natural home in the five layers above, that's a signal to think about where it actually belongs before writing throwaway test code around it.

## Adding a new test — the pattern

1. Confirm the function/constant you need is exported (bulk-exported already as of this scaffolding — if something new gets added to `check-jobs.ts` later, export it explicitly).
2. Pick the layer by what's actually being tested, using the five definitions above — not by what feels closest to a shenny file name.
3. If it needs real state on disk, use the `process.chdir()` + `mkdtempSync` pattern from `tests/state/tracker.test.ts`. Don't mock `fs` for this project's state layer — real files are what actually broke before, so real files are what should catch it again.
4. If it needs a fixture HTTP response, check `tests/fixtures/` first — nearly every source now has real captured data (Statheros, Adzuna, Quarterhill, TherapyNotes, Workable, Greenhouse, Ashby, RemoteOK, USAJOBS, SOLTECH) rather than invented placeholders; only Lever's is built from its documented public API shape, for lack of a live configured board. Prefer extending those over fabricating new ones from scratch.
5. Run it for real before considering it done. Every test in this suite — all 198 Vitest tests and all 5 Playwright tests — was actually executed and confirmed passing. That standard doesn't lower going forward.
